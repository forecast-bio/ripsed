# Benchmarks

Comparative benchmarks of ripsed against GNU sed, [sd](https://github.com/chmln/sd),
and `perl -pi`, measured with [hyperfine](https://github.com/sharkdp/hyperfine).

## Reproducing

Everything below is generated by a single command — the corpus generator,
tool invocations, and measurement parameters live in
[`xtask/src/main.rs`](xtask/src/main.rs):

```bash
cargo xtask bench-compare           # full corpus (~minutes)
cargo xtask bench-compare --quick   # small corpus sanity run
```

Requirements: `hyperfine` on PATH; `sed`, `sd`, and `perl` are each
optional and skipped when absent. The harness builds `ripsed` in release
mode from the current checkout, so numbers always reflect HEAD.

## Methodology

- Every tool performs a functionally equivalent **in-place** edit.
- A pristine corpus is copied fresh before **every** timing iteration
  (`hyperfine --prepare`), so no tool ever sees another's output and
  later iterations don't degrade.
- `sed`, `sd`, and `perl` have no recursive file discovery, so tree
  scenarios feed them an explicit file list via
  `find -name '*.txt' -print0 | xargs -0`. ripsed discovers files
  itself — its parallel, gitignore-aware discovery is part of what is
  being measured, not overhead to factor out.
- ripsed additionally writes its undo log during the timed run (default
  behavior; none of the other tools offer undo). That cost is included —
  these are out-of-the-box comparisons, not tuned ones.
- Scenarios:
  - **tree-replace** — literal find/replace across a tree of
    1000 files × 200 lines (~10% of lines match), 20 subdirectories.
  - **tree-no-match** — identical tree, pattern matches nothing
    (exercises ripsed's whole-buffer prescreen).
  - **big-file** — one 64 MiB file, ~50% of lines match.

## Results

Taken 2026-06-09 on WSL2 (Linux 6.6), Rust 1.96.0, ripsed `de5052f`,
GNU sed 4.9, sd 1.0, perl 5.40. Corpus: 1000 files × 200 lines (tree
scenarios), 64 MiB (big-file).

### tree-replace — literal replace across a source tree

| tool | mean | min |
|---|---|---|
| ripsed | 471.8 ms | 110.7 ms |
| sed | 279.7 ms | 134.1 ms |
| sd | 246.2 ms | 192.0 ms |
| perl **(fastest)** | 237.3 ms | 130.8 ms |

### tree-no-match — pattern matches nothing in the tree

| tool | mean | min |
|---|---|---|
| ripsed **(fastest)** | 176.7 ms | 140.3 ms |
| sed | 260.6 ms | 107.4 ms |
| sd | 240.1 ms | 132.1 ms |
| perl | 263.7 ms | 204.5 ms |

### big-file — literal replace in one large file

| tool | mean | min |
|---|---|---|
| ripsed | 6271.6 ms | 5515.9 ms |
| sed **(fastest)** | 993.7 ms | 457.7 ms |
| sd | 1165.8 ms | 499.5 ms |
| perl | 1636.7 ms | 689.8 ms |

### Honest reading

- **tree-no-match**: ripsed wins — the whole-buffer prescreen rejects
  non-matching files at substring-search speed while the others run
  their full edit machinery per file.
- **tree-replace**: ripsed loses on mean (high variance under WSL2 —
  its *min* is the best of the four) and does strictly more work per
  file than the others: undo-log recording and atomic temp-file+rename
  writes are always on.
- **big-file**: ripsed loses clearly. Since these numbers were taken,
  undo recording is capped (`undo.max_file_bytes`, default 4 MiB —
  oversize files are edited but not recorded) and `--no-undo` exists;
  measured on the same corpus that trims roughly a second off the
  mean. The remaining gap is structural: the line-oriented engine
  materializes the file as separate line `String`s before rejoining,
  while sed streams. For very large single files, `sed` remains the
  right tool today.

## Caveats

- Single machine, single OS; absolute numbers will differ on your
  hardware. Relative ordering is the interesting part.
- These runs were taken under WSL2, where filesystem-heavy workloads
  (the tree scenarios' per-iteration corpus copies and many small
  writes) carry more OS overhead and variance than native Linux. Treat
  small deltas as noise; rerun with `cargo xtask bench-compare` on your
  own machine before drawing fine-grained conclusions.
- `hyperfine` reports statistical outliers on busy systems; the harness
  uses a warmup run and lets hyperfine pick the run count (minimum 10).
